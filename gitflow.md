# Git Flow

## Git Flow란?

Git Flow는 Git을 사용하는 프로젝트에서 브랜치를 체계적으로 관리하기 위한 방법이다.
프로젝트 진행 과정에서 개발, 테스트, 배포, 버그 수정 등의 작업을 각각 다른 브랜치에서 진행하도록 구조를 정리한 전략이다.

## Git Flow의 주요 브랜치

- **main**

프로젝트의 최종 배포 버전이 저장되는 브랜치이다.
항상 안정적인 상태의 코드가 유지되어야 하며 실제 서비스에 사용되는 코드가 존재한다.

- **develop**

개발 과정에서 기능을 통합하는 중심 브랜치이다.
여러 기능 브랜치에서 작업이 끝나면 이 브랜치에 합쳐지고 테스트가 진행된다.

- **feature**

새로운 기능을 개발할 때 사용하는 브랜치이다.
보통 develop 브랜치에서 생성하여 작업한 뒤 완료되면 다시 develop으로 병합한다.

- **release**

배포를 준비하는 단계에서 사용하는 브랜치이다.
개발이 완료된 후 develop에서 만들어지며, 마지막 점검과 수정이 이루어진다.

- **hotfix**

이미 배포된 서비스에서 긴급하게 수정해야 하는 문제가 발생했을 때 사용하는 브랜치이다.
main 브랜치에서 생성하여 수정 후 main과 develop에 반영한다.

## Git Flow 작업 흐름

일반적인 작업 과정은 다음과 같다.

1. 기능 개발

develop 브랜치에서 feature 브랜치를 생성하여 새로운 기능을 개발한다.
작업이 끝나면 다시 develop 브랜치에 병합한다.

2. 배포 준비

새로운 버전을 배포하기 전에 develop에서 release 브랜치를 만들어 테스트와 최종 수정 작업을 진행한다.

3. 배포 진행

release 브랜치를 main 브랜치에 병합하고 버전 태그를 생성하여 배포를 진행한다.
이 변경 사항은 다시 develop에도 반영된다.

4. 긴급 수정

서비스 운영 중 문제가 발생하면 main에서 hotfix 브랜치를 만들어 수정한 후 main과 develop에 반영한다.

## Git Flow의 특징

- **장점**

브랜치 역할이 명확하여 협업 시 관리가 쉽다.

배포 과정이 체계적으로 정리된다.

큰 규모의 프로젝트에서 안정적인 개발이 가능하다.

- **단점**

브랜치 구조가 많아 관리가 복잡할 수 있다.

작은 프로젝트나 빠른 배포가 필요한 서비스에서는 다소 비효율적일 수 있다.

## GitHub Flow

GitHub Flow는 Git Flow보다 단순한 브랜치 전략이다.main 브랜치를 기준으로 새로운 브랜치를 만들어 작업한 뒤 Pull Request를 통해 코드 리뷰 후 바로 병합하고 배포하는 방식이다.
이 방법은 지속적인 배포(Continuous Delivery) 환경에서 많이 사용된다.
